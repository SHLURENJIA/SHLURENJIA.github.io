写出两种单例模式，分析其优劣

单例模式（singleton ），一种很基础的设计模式，在面试时候经常会要求手写不同的单例模式。主要有三种：饿汉式，懒汉式，双重锁模式。

单例模式主要要满足一下特点：

  • 一个类只有一个实例，这是最基本的
  • 它必须自行创建这个实例
  • 它必须自行向整个系统提供这个实例

饿汉式

类在加载的时候就完成了初始化，所以类加载比较慢，但是获取对象的速度较快。

饿汉模式，因为比较饿，所以很着急，就在类加载的时候赶紧就创建实例了。

优劣分析：

好处： 简单明了，不用在意线程问题

缺点：

  • 类加载较慢
  • 无法延迟实例化。在大环境中如果使用了过多的饿汉单例，无论是否使用到都会加载，产生过多实例对象，并且加载时间很长。

###懒汉式

很懒，所以在类加载的时候不创建实例，因此类加载速度较快，但是运行时获取实例较慢

优劣分析：

优点： 类加载快，线程安全

缺点：

  • 效率低，每次调用实例都要判断同步锁。

PS：网上找到有些懒汉式是没有用 synchronized 修饰了 get 方法，是线程不安全的懒汉式。优缺点很明显，他的获取对象实例变快了，但是在多线程中不安全。

双重锁模式

懒汉式的变种

是饱汉模式的优化，进行双重判断，当已经创建过实例对象后就无需加锁，提高效率。也是一种推荐使用的方式。

但是，据说这种模式，在并发量不大，安全性不太高的情况下能完美运行，但是 instance = new instance()这句，在JVM编译的过程中会出现指令重排的问题，这就会导致当 instance实际上还没初始化，就可能被分配了内存空间，也就是说会出现 instance !=null 但是又没初始化的情况，这样就会导致返回的 instance 不完整

不太懂，待验证。

内部类实现方式

内部类实现方式，可以推荐使用。延迟加载，线程安全（java中class加载时互斥），减少内存消耗

枚举方法

据说是网上推荐使用的方法，但是使用貌似不太广泛。

1.从Java1.5开始支持;

2.无偿提供序列化机制;

3.绝对防止多次实例化，即使在面对复杂的序列化或者反射攻击的时候;
